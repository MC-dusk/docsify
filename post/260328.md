## Python 系统代理网络连接 Bug

这是一个**贯穿 Python 3.7 到 3.10 早期版本、影响所有依赖系统代理联网工具**的经典底层 Bug，直接导致 pip、requests、urllib、conda 等联网功能在开启系统代理时全部失效，也是国内开发者使用代理工具（Clash、V2Ray、Netch、企业内网代理）时最常遇到的网络故障之一。

---

### 一、Bug 核心现象
- 电脑配置了**系统代理**（Windows 代理设置 / macOS/Linux 系统代理）
- 浏览器、其他软件联网正常
- 但 Python 相关工具全部无法联网：
  - `pip install` 报 `ProxyError`、`Tunnel connection failed`、`Connection refused`
  - `requests.get()`、`urllib.request` 直接连接失败
  - 关闭系统代理后立刻恢复正常

---

### 二、根本原因

Windows/macOS 系统代理配置格式为：
```
http=127.0.0.1:7890; https=127.0.0.1:7890
```

几乎所有代理软件**只提供 HTTP 代理端口**，只接受：
```
http://127.0.0.1:7890
```

而 Python 在读取系统代理时，**错误地为 `https` 协议拼接了 `https://` 前缀**：
```
https://127.0.0.1:7890
```

代理服务器不支持 HTTPS 直接连接，导致握手失败、连接被拒绝，最终表现为“Python 联网失败”。

---

### 三、受影响版本范围

这个问题本质是**同一个底层逻辑错误**，只是在不同版本分支中修复策略不同：

#### 1. Python 3.7.x、3.8.x（全系列，含 3.8.5）

- **永久存在此 Bug**
- 官方**不再向后移植修复**
- 无官方补丁，**必须手动修改源码**才能彻底修复
- 这也是很多老教程提到“要改 urllib.request.py”的来源

#### 2. Python 3.9.0 ~ 3.9.18

- 同样存在此代理解析错误
- 官方在后续小版本中修复
- 升级到 **3.9.19 及以上** 即可解决

#### 3. Python 3.10.0 ~ 3.10.11

- 与 3.9 同源 Bug
- 升级到 **3.10.12 及以上** 修复

#### 4. Python 3.11+

- 出厂已完全修复，无此问题

---

### 四、为什么只影响 Python，不影响浏览器/其他软件

- 浏览器、IDE、系统网络栈会**自动将 HTTPS 代理按 HTTP 协议访问**
- Python 早期版本严格按协议名拼接前缀，**没有做兼容处理**
- 代理工具普遍只实现 HTTP CONNECT 隧道，不提供 HTTPS 代理服务

---

### 五、修复方案（按推荐优先级）

#### 1. 升级 Python（最稳妥）

- 3.9 → 升级到 3.9.19+
- 3.10 → 升级到 3.10.12+
- 直接使用 3.11 / 3.12 / 3.13 新版

#### 2. 手动修改源码（3.7 / 3.8 专用）

修改文件：`Lib/urllib/request.py`
在系统代理解析逻辑中，强制将 `https` 代理前缀改为 `http://`。

#### 3. 临时环境变量覆盖（不改源码、不升级）

在终端执行，强制指定正确代理协议：
```bash
# Windows
set HTTP_PROXY=http://127.0.0.1:7890
set HTTPS_PROXY=http://127.0.0.1:7890

# macOS/Linux
export HTTP_PROXY=http://127.0.0.1:7890
export HTTPS_PROXY=http://127.0.0.1:7890
```

#### 4. pip 命令行指定代理

```bash
pip install 包名 --proxy http://127.0.0.1:7890
```

---

### 六、总结

**Python 3.7～3.10 早期的系统代理联网失败，本质是同一个底层 Bug：**
> 读取系统代理时，错误为 HTTPS 协议添加 `https://` 前缀，导致无法连接仅支持 HTTP 的代理端口。

- 3.7/3.8：官方不修复，只能改源码或临时绕过
- 3.9/3.10：后期小版本修复，升级即可
- 3.11+：完全无此问题

这是 Python 在 Windows/macOS 平台上**历史最久、影响最广**的网络代理兼容性问题之一。
